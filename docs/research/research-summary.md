# [종합 분석] OTA 숙박 플랫폼 리서치 요약

> `user-research.md`의 플랫폼별 리서치 내용을 바탕으로 종합 분석한 문서.

## 1. 플랫폼별 특징 요약

| 항목 | 국내A | 국내B | 해외A |
|------|-------|-------|-------|
| 상품군 | 해외숙소, 호텔/리조트, 펜션/풀빌라, 모텔 (타입별 분리) | 해외숙소, 국내숙소 (국내는 단일 상품군으로 통합) | 숙소, 프라이빗숙소 |
| 숙소유형 구분 | 국내 3개 타입 + 해외 1개, 타입별 UI/필터 상이 | 국내 모든 유형 하나로 통합, 필터로 분기 | 숙소/프라이빗 2분류 |
| 객실 옵션 | 객실당 복수 옵션 (호텔/해외), 단일 옵션 (펜션/모텔) | 국내 단일, 해외 복수 옵션 | 객실당 복수 옵션 (조식·취소정책 조합) |
| 리뷰 별점 | 0.0 ~ 5.0 | 0.0 ~ 10.0 | 카테고리별 평점 (위치·청결·서비스 등) |
| 리뷰 특징 | 포토후기 (국내), 키워드 태그 (호텔) | 작성자 신뢰도 강조 (리뷰수·사진수 등) | 후기 요약, 검색어별 후기, 도움됨 투표 |
| 예약유형 | 모텔만 대실/숙박 구분 | 모텔만 대실/숙박 구분 (30분 단위) | 숙박만 |
| 외부 연동 | 해외숙소 가격 조회 시 외부 API 통신 | 해외 객실 조회 시 외부 연동 | 자체 재고 |
| 쿠폰 | 숙소 단위 1:1 / 1:N | 최대 2개 중복, 상품권 적용 가능 | 할인코드 방식 |
| 지도 | 금액 마커로 표기 | 필터+리스트+지도 연동 | 도심 거리 기준 표기 |
| 결제 | 선불 | 선불 | 선불/후불 |
| 성급/등급 | 호텔 성급 인증 (국내) | 등급 필터 존재 | 숙소 등급 필터 |
| 숙소 이벤트 | 호텔/리조트에 존재 | 복수 등록 가능, 정형화된 포맷 | 없음 |
| 다국어/통화 | 미확인 | 미확인 | 다국어·다통화 지원 |
| 예약 시간 제한 | 없음 | 없음 | 조회 후 일정 시간 내 결제 필요 (hold) |

## 2. 공통점

| # | 공통 기능 | 설명 |
|---|----------|------|
| 1 | 검색 조건 | 지역/검색어 + 일정 + 인원(+객실) 조합 조회 |
| 2 | 필터 | 가격 범위, 숙소유형, 시설/서비스, 예약 가능 여부 등 |
| 3 | 정렬 | 추천순, 가격순, 평점순 |
| 4 | 일정 기반 가격 | 선택한 박수에 맞는 총 가격 표시 |
| 5 | 지도 기능 | 위도/경도 기반 숙소 위치 표시 |
| 6 | 이미지 관리 | 숙소 대표사진 + 객실별 사진, 카테고리 분류 |
| 7 | 찜(위시리스트) | 숙소 단위 찜 기능 |
| 8 | 쿠폰/할인 | 숙소 단위 쿠폰 적용 (1:1 또는 1:N) |
| 9 | 리뷰 | 별점 + 정렬 + 객실별 조회 |
| 10 | 객실 구조 | 숙소 > 객실 > 옵션(요금제) 계층 구조 |
| 11 | 예약자 정보 | 이름, 전화번호, 이메일 등 기본 투숙객 정보 입력 |
| 12 | 요청사항 | 체크인 시간, 침대 구성, 기타 자유 입력 |
| 13 | 취소/환불 규정 | 예약 전 취소 정책 고지 |
| 14 | 약관 동의 | 결제 전 필수 약관 동의 |

## 3. 차이점

| # | 항목 | 차이 내용 |
|---|------|----------|
| 1 | 숙소유형 관리 | 국내A는 타입별 완전 분리(필터/카테고리 상이), 국내B는 단일 상품군에 필터로 분기 |
| 2 | 객실-옵션 관계 | 호텔/해외는 1객실:N옵션, 펜션/모텔은 1객실:1옵션 — 통합 모델링 필요 |
| 3 | 별점 체계 | 5점제(국내A) vs 10점제(국내B) vs 카테고리별 세분화(해외A) |
| 4 | 대실(시간제) | 국내 모텔만 대실 지원, 30분 단위 시간 슬롯 예약 |
| 5 | 리뷰 깊이 | 국내A: 키워드 태그, 국내B: 작성자 신뢰도, 해외A: 카테고리 평점+검색+도움됨 |
| 6 | 외부 연동 | 해외숙소 가격/객실 조회 시 외부 Supplier API 호출 (국내A·B 공통) |
| 7 | 결제 방식 | 국내는 선불 중심, 해외A는 후불 옵션 존재 |
| 8 | 예약 hold | 해외A만 조회 후 일정 시간 내 결제 필요 (재고 임시 확보) |
| 9 | 다국어/통화 | 해외A만 다국어·다통화 지원 |
| 10 | 사진 카테고리 | 숙소 타입별로 사진 카테고리가 다름 (해외숙소 전용, 호텔 전용 등) |

## 4. 구현 대상 리스트업

### 4.1. 핵심 구현 (Core)
| # | 기능 | 우선순위 | 비고                                                               |
|---|------|---------|------------------------------------------------------------------|
| 1 | 숙소 CRUD | 높음 | 숙소유형(호텔/펜션/모텔 등) 구분, 기본정보·위치(위도/경도)·시설/서비스                       |
| 2 | 객실 CRUD | 높음 | 숙소 하위, 객실별 이미지·편의시설 관리                                           |
| 3 | 객실 옵션(요금제) | 높음 | 1객실:N옵션 구조 (조식 유/무, 취소정책 등 조합)                                   |
| 4 | 재고(Inventory) 관리 | 높음 | 날짜별, 시간별 객실 재고, 동시성 제어(오버부킹 방지)                                  |
| 5 | 요금 관리 | 높음 | 날짜별 요금 설정, 박수 기반 합산 가격 계산 (시간별 금액이 달라야 하는지 고려)                   |
| 6 | 숙소 검색 | 높음 | 지역+일정+인원 조합, 필터(가격·유형·시설), 정렬(추천·가격·평점)                          |
| 7 | 예약 프로세스 | 높음 | 예약자 정보 입력 -> 재고 차감 -> 예약 확정, 동시성 처리 (숙소설정에 따라서 자동확정처리 or 수동확정처리) |
| 8 | 예약 취소 | 높음 | 취소 정책 기반 환불 규칙, 재고 복구                                            |
| 9 | 이미지 관리 | 중간 | 숙소/객실 단위 이미지 리사이징 후 등록, 카테고리 분류                                  |
| 10 | 외부 Supplier 연동 | 중간 | 어댑터 패턴으로 외부 공급사 가격/객실 통합                                         |
| 11 | 대실(시간제 예약) | 중간 | 모텔 30분 단위 슬롯, 숙박/대실 예약유형 구분, 재고 모델 확장                            |
| 12 | 예약 hold(임시 확보) | 중간 | 조회 시 재고 임시 확보, TTL 내 미결제 시 자동 복구 (오버부킹 방지)                       |
| 13 | 다국어 지원 | 중간 | `_translation` 테이블 패턴으로 초기부터 다국어 구조 설계 (숙소명, 객실명, 설명 등)          |

### 4.2. 설계 중심 (ERD + API 설계, 메서드 명칭 정도 내부 로직은 주석으로 구현 방안 제시 정도, 최소 구현)
| # | 기능 | 비고 |
|---|------|------|
| 1 | 회원 | 예약에 필요한 최소 데이터 (이름, 연락처, 이메일) |
| 2 | 쿠폰 | 숙소 단위 1:1/1:N 매핑, 적용/검증 로직 |
| 3 | 결제 | 결제 수단 선택, 결제 상태 관리 |
| 4 | 리뷰 | 별점 + 텍스트, 정렬, 객실별 조회 |
| 5 | 지도 검색 | 위도/경도 기반 반경 검색, 금액 마커 |
| 6 | 찜(위시리스트) | 회원-숙소 매핑 |
| 7 | 숙소 이벤트/프로모션 | 숙소 단위 이벤트 등록 (복수) |
| 8 | 다통화 | 통화 변환, 환율 적용 (다국어와 별도) |
| 9 | 리뷰 고도화 | 포토리뷰, 카테고리별 평점, 키워드 태그, 작성자 신뢰도 |
