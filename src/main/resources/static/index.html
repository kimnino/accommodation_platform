핵심 설계 결정

01

채널별 완전 분리

admin / extranet / customer 각 채널이 독립적인 Inbound Port와 UseCase를 갖습니다. 채널 간 직접 참조는 컴파일 레벨에서 금지되어 있으며, 공유 로직만 core 패키지로 올라갑니다.

02

수정 승인 구조

파트너가 숙소 정보를 변경하면 즉시 반영되지 않고 ModificationRequest 레코드가 생성됩니다. 관리자가 승인하는 순간 실제 데이터에 반영되며, 변경 데이터는 JSON으로 저장해 유연성을 확보합니다.

03

숙박 vs 대실 타입 분리

재고와 예약 모두 STAY / HOURLY 두 타입으로 분리됩니다. 대실은 시간 슬롯 단위로 재고가 관리되며, 가격 설정·예약 API 엔드포인트도 별도로 설계되어 있습니다.

04

예약 동시성 처리

동일 재고에 동시 예약 요청이 들어올 때 오버부킹을 방지합니다. Pessimistic Lock(DB) 또는 Redis Distributed Lock을 통해 원자성을 보장하며, CountDownLatch 기반 동시성 테스트로 검증합니다.

05

공급사 연동 어댑터

외부 OTA 공급사(Expedia, Agoda 등)를 SupplierClient Outbound Port로 추상화합니다. 공급사별 구현체는 어댑터 레이어에 격리되며, 공급사 전용 태그 그룹으로 데이터를 구분합니다.

06

도메인 순수성 유지

엔티티와 VO는 순수 Java로 작성되며 JPA / Spring 어노테이션이 없습니다. 도메인 서비스는 @Service 대신 DomainServiceConfig에서 @Bean으로 등록해 프레임워크 의존성을 분리합니다.
