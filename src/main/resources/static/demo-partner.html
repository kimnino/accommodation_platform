숙소 등록

왜 이렇게 설계했나: Extranet 채널은 파트너가 직접 숙소를 등록합니다. 등록 직후 상태는 PENDING이며, 관리자 승인 후 ACTIVE로 전환됩니다. 채널 분리 원칙에 따라 extranet/{domain}/adapter/in/web 하위 Controller만 이 API를 담당합니다.
POST /api/v1/extranet/accommodations X-Partner-Id: {partnerId}
버튼을 눌러 API를 호출하세요
PENDING 관리자 승인 대기 중

숙소가 등록되었으나 관리자 승인 전까지 고객에게 노출되지 않습니다. 관리자 채널에서 승인하면 ACTIVE 상태로 전환됩니다. 이 구조 덕분에 불완전하거나 허위 숙소가 자동으로 노출되는 것을 방지합니다.

객실 등록

왜 이렇게 설계했나: 숙소(Accommodation) → 객실(Room) → 객실 옵션(RoomOption) 3단계 계층을 갖습니다. 옵션이 실제 예약·재고의 단위입니다. 각 옵션에 숙박/대실 추가 금액을 독립적으로 설정해 동일 객실에서 두 타입을 모두 판매할 수 있습니다.

숙소 목록이 없으면 Step 1에서 먼저 숙소를 등록하거나 새로고침을 누르세요.

객실 옵션 설정 (재고·예약의 실제 단위)

POST /api/v1/extranet/accommodations/{id}/rooms
이어서 →
POST /api/v1/extranet/accommodations/{accId}/rooms/{roomId}/options

재고 & 가격 일괄 설정

왜 이렇게 설계했나: 숙박(STAY)과 대실(HOURLY)은 완전히 다른 재고 단위로 관리됩니다. 숙박은 날짜 단위, 대실은 시간 슬롯 단위입니다. 가격 설정도 분리되어 있어 동일 객실 옵션에서 두 타입의 가격을 독립적으로 운영합니다.

숙소 선택 → 객실 옵션 자동 로드. 목록에 없으면 Step 2에서 객실을 먼저 등록하세요.

POST /api/v1/extranet/room-options/{id}/inventories
이어서 →
POST /api/v1/extranet/room-options/{id}/prices
버튼을 눌러 API를 호출하세요
숙박 vs 대실 설계 포인트
숙박: check_in_date ~ check_out_date (LocalDate 기반)
대실: date + start_time (LocalTime 기반 — 숙소 현지 시간)
시스템 타임스탬프(created_at 등)만 Instant(UTC)로 저장합니다.

숙소 정보 수정 요청

왜 이렇게 설계했나: 파트너가 숙소 정보를 직접 수정하면 즉시 반영되지 않습니다. 수정 요청(ModificationRequest)이 생성되고 관리자가 승인할 때만 실제 데이터에 반영됩니다. 이 구조로 브랜드 무결성과 고객 신뢰를 보호합니다. 변경 데이터는 JSON으로 저장해 유연성을 확보합니다.

수정할 숙소를 선택하세요. 목록에 없으면 새로고침을 누르세요.

PUT /api/v1/extranet/accommodations/{id}

승인 대기 수정 요청이 접수되었습니다

파트너의 수정 요청은 즉시 반영되지 않으며 관리자 승인이 필요합니다.
변경 데이터는 JSON 형태로 modification_requests 테이블에 저장됩니다.
관리자가 승인하면 실제 숙소 데이터에 반영되고, 거절 시 사유와 함께 반려됩니다.

내 숙소 목록 조회

왜 이렇게 설계했나: Extranet 채널은 X-Partner-Id 헤더로 파트너를 식별합니다. 각 파트너는 자신의 숙소만 조회·관리할 수 있습니다. 채널 분리 덕분에 관리자가 전체 숙소를 보는 API와 파트너가 자신의 숙소를 보는 API가 완전히 분리되어 있습니다.
GET /api/v1/extranet/accommodations X-Partner-Id: {partnerId}
