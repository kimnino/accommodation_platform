숙소 검색

왜 이렇게 설계했나: 검색 쿼리는 QueryDSL로 동적 생성됩니다. JPQL 문자열 조합은 금지되어 있습니다. 가격 필터는 Java 후처리가 아닌 WHERE절에서 처리해 페이징 정확성을 보장합니다. 2단계 배치 조회(ID 목록 → IN절)로 최저가와 이미지를 효율적으로 조회합니다.
GET /api/v1/accommodations
검색 버튼을 눌러 API를 호출하세요

숙박 예약 (STAY)

왜 이렇게 설계했나: 숙박 예약은 check_in_date ~ check_out_date (LocalDate) 기반입니다. 예약 성공 시 상태는 PAYMENT_WAITING으로 시작되며 10분 내 결제가 필요합니다. reservation_request_id(UUID)로 멱등성을 보장해 중복 예약을 방지합니다.

Step 1 검색 후 숙소를 클릭하면 자동 선택됩니다. 객실 옵션은 숙소 선택 후 자동 로드됩니다.

POST /api/v1/reservations/stay X-Member-Id: {memberId}
버튼을 눌러 API를 호출하세요
예약 상태 흐름:
PAYMENT_WAITING → (결제 완료) → CONFIRMED → (체크인) → COMPLETED
결제 대기 중 취소 시: CANCELLED / 결제 시간 초과 시 자동 만료

대실 예약 (HOURLY)

왜 이렇게 설계했나: 대실 예약은 date + start_time + end_time (LocalTime) 기반입니다. 시간은 숙소 현지 시간 기준으로 처리됩니다 (14:00 입실 = 해당 숙소 현지 14시). 숙박과 완전히 분리된 API 엔드포인트로 설계해 두 타입의 비즈니스 로직이 섞이지 않습니다.

숙소 선택 후 객실 옵션이 자동 로드됩니다.

POST /api/v1/reservations/hourly X-Member-Id: {memberId}
버튼을 눌러 API를 호출하세요
숙박 vs 대실 타입 비교:
숙박: LocalDate 체크인/체크아웃 — 날짜 단위 재고
대실: LocalDate 날짜 + LocalTime 시작시간 — 시간 슬롯 단위 재고
두 타입은 동일한 객실 옵션에서 독립적으로 운영됩니다.

예약 동시성 처리 시뮬레이션

왜 이렇게 설계했나: 동일 재고에 여러 요청이 동시에 들어올 때 오버부킹이 발생하면 안 됩니다. Pessimistic Lock(DB) 또는 Redis Distributed Lock으로 원자성을 보장합니다. 실제 코드에서는 CountDownLatch + ExecutorService로 동시성 테스트를 작성합니다. 아래 버튼으로 동시에 N개의 예약 요청을 보내 하나만 성공하는 것을 확인할 수 있습니다.

동시 예약 요청 시뮬레이션

재고가 1개인 객실에 N개의 예약 요청을 동시에 전송합니다.
정상 동작 시: 1개만 성공(SUCCESS), 나머지는 재고 부족으로 실패(INVENTORY_NOT_AVAILABLE)

실행 버튼을 누르면 동시 예약 요청이 시작됩니다...
// 실제 동시성 테스트 코드 (JUnit 5)
@Test
void 동시_예약_요청_시_하나만_성공해야_한다() {
  var latch = new CountDownLatch(N);
  var executor = Executors.newFixedThreadPool(N);
  var successCount = new AtomicInteger(0);
  // ... N개 동시 실행 후
  assertThat(successCount.get()).isEqualTo(1);
}
